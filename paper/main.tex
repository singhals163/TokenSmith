\begin{abstract}
TokenSmith is a local-first RAG system for textbook question answering.
The original implementation is fully functional but is impractical for
a new user trying it on a fresh textbook: cold ingestion before the
first answer takes close to three hours on a server, and longer on a
laptop. We re-target the system for consumer hardware (one GPU with at
most 8\,GiB of memory, four CPU cores) by making four pipeline changes:
parallel PDF extraction, a \texttt{pypdfium2} fast extraction path, a
hardware-aware embedding router with length-sorted batches, and a
pluggable embedder that can target \texttt{llama.cpp}, GPT4All (CUDA or
Vulkan), or HuggingFace \texttt{sentence-transformers} at a fixed call
site. On a Quadro RTX 6000 capped at 8\,GiB, every variant stays well
under the cap. A 23\,M MiniLM embedder finishes the end-to-end pytest
suite in 40 seconds at a final score of 0.578, within 1.4 points of
the 4\,B baseline (0.592) which itself fits in 5.4\,GiB. \shivam{Can you mention the most efficient extraction time and embedding time, and then if you have a metric say that the full run on a consumer grade laptop from ingesting to the generation of first response would take approx x seconds on our framework. \texttt{Qwen3-Embedding} model; retrieval and generation take seconds\shivam{Add the exact numbers}.
A new user pointing TokenSmith at a fresh textbook therefore waits
hours before the first answer comes back. On a 4-core laptop the same
extractor runs even longer \shivam{Do we have an estimate of run time with these resources, you can simply add the baseline extraction time (the one with single core), + indexing time, + generation time on RTX6000 gpu with llama model}. The system works, but in practice no
student would sit through a first run.

The goal of this project is to make ingestion fast enough that a
student can drop a new PDF in and get a useful answer in the same
study session, without sending data off the host. We use the
Silberschatz textbook as the canonical corpus (2{,}195 pages, 2{,}664
chunks after section-aware splitting) and target a consumer machine
with at most 8\,GiB of GPU memory and four CPU cores. Our most
efficient configuration peaks at roughly 3\,GiB of host RAM and 2\,GiB
of GPU memory during a full ingestion; the heaviest variant (the 4\,B
GGUF embedder plus the 1.5\,B generator, both on GPU) peaks at
5.4\,GiB of GPU memory. The 32\,GiB host allocation we use during
evaluation is over-provisioned, and a typical consumer laptop with
16\,GiB of host RAM is sufficient.

The pipeline has four stages: PDF extraction to markdown, indexing
into FAISS and BM25, retrieval with a cross-encoder reranker, and
generation through a 1.5\,B Qwen2.5-Instruct GGUF model on
\texttt{llama.cpp}. We make one change per stage and measure the
impact on a server (A100 / V100) and on a Quadro RTX 6000 used as the
consumer-GPU proxy.

\section{Phase 1: Faster PDF Extraction}

The single largest contributor to time-to-first-token is PDF
extraction. We attack it twice:\shivam{get rid of colon} first by parallelizing the existing
\texttt{docling} backend across CPU workers, then by replacing the
backend itself for plain-text textbooks where layout analysis is
unnecessary.

\subsection{Map-Reduce Architecture for Extraction}

The original extractor calls \texttt{docling} on the full PDF
sequentially. Page-level layout analysis is the slow part of
\texttt{docling}, and it is independent across pages, so we restructure
the extractor as a map-reduce pipeline.

In the map phase, \texttt{split\_pdf} divides the master PDF into
fixed-size page-range chunks using \texttt{pypdf} and writes each chunk
to a temporary file. Each chunk is dispatched to a worker process
(\texttt{process\_pdf\_chunk}) that loads its own \texttt{docling}
instance, converts the chunk, and serializes it page by page. A
\texttt{ProcessPoolExecutor} runs the workers; results come back keyed
by start-offset. In the reduce phase, results are sorted on offset and
concatenated into a single markdown document. The \texttt{docling}
import is deferred into the worker function so that downstream callers
of the extraction module (the indexer, the pytest harness) do not pay
its 100\,MB cold-load cost when they only need the markdown section
helpers.

We swept the worker count on a 64-core server in
$\{1, 2, 4, 8, 16, 32, 62\}$. Table~\ref{tab:phase1} reports the
total wall time per configuration and Figure~\ref{fig:extraction-scaling}
plots the same data on a log y-axis.

interactive session. The next subsection attacks the extractor itself.

\subsection{pypdfium2 Fast Path}

Even fully parallel, \texttt{docling} is overkill for plain-text
textbooks. The Silberschatz textbook has predictable numbered section
headings and few tables; layout analysis costs minutes per pass and
gives us nothing we cannot recover with a regex.

The new module \texttt{src/preprocessing/pypdfium\_extractor.py} uses
\texttt{pypdfium2} to read raw text from each page range. There is no
temporary split; workers seek into the master PDF directly. A short
post-processor strips table-of-contents lines and promotes lines that
look like ``\texttt{N.N Title}'' to ``\texttt{\#\# N.N Title}'' so the
existing markdown section splitter still works. The \texttt{docling}
backend remains available for documents that need real layout
analysis.

On the RTX 6000 host (4 CPU cores), full extraction drops from
2{,}018\,s with \texttt{docling} (the 4-worker number above) to
8.66\,s with \texttt{pypdfium2}, a roughly 230$\times$ \shivam{highlight the numbers in these paragraph, they're real massive win} improvement on
this hardware. End-to-end final score on the regenerated index is
0.606, slightly above the 0.592 \shivam{This is second massive win} we get with the \texttt{docling}
pipeline; the regex-based heading promotion recovers section
boundaries reliably without the layout-analysis pass.

\section{Phase 2: Hardware Router and Batch Sorting}

Two small changes in the embedder reduce ingestion wall time.

\paragraph{Length-sorted batches.}
\texttt{llama.cpp} pads each embedding batch to the longest text in
that batch. Random ordering wastes compute on short texts that share a
batch with one long text. We sort inputs by length descending before
batching and permute the result back after encoding. The FAISS index
ID order is unchanged.

\paragraph{Dynamic GPU / CPU routing.}
\texttt{detect\_gpu()} checks for \texttt{nvidia-smi} (CUDA) or Apple
Silicon (Metal). When a GPU is reachable the indexer takes a sequential
path with \texttt{batch\_size=32}; otherwise it uses a CPU
\texttt{multiprocessing.Pool} with the \texttt{mmap}'d GGUF shared
across workers. This avoids a CPU pool fighting the GPU's allocator on
mixed hosts.

\section{Phase 3: Pluggable Embedding Backends}

A 4\,B embedder is heavy. Phase 3 makes the embedder swappable so we
can compare lighter alternatives at a fixed call site.

\texttt{src/embedder.py} introduces a \texttt{BaseEmbedder} ABC with
three implementations: \texttt{LlamaCppEmbedder} for GGUF models,
\texttt{GPT4AllEmbedder} for Nomic GGUFs (with \texttt{device=kompute}
for Vulkan or \texttt{device=cuda} for NVIDIA), and
\texttt{HFSentenceTransformerEmbedder} for HuggingFace
\texttt{sentence-transformers}. A factory
\texttt{make\_embedder(model\_path, backend)} returns the right
implementation and a back-compat alias keeps existing import sites
working.

Two adjacent fixes ship in this phase. We add a sentence-level BLEU-4
metric with NIST method-1 smoothing under \texttt{tests/metrics/bleu.py}.
We also fix a pollution bug in \texttt{tests/metrics/semantic.py}: the
original code set \texttt{CUDA\_VISIBLE\_DEVICES=''} at module import,
which silently disabled the GPU for any sibling process in the same
interpreter. We pin the scoring model to \texttt{device='cpu'} instead.
End-to-end pytest wall on a typical V100 run drops from $\sim$700\,s to
$\sim$45\,s.

\section{Consumer-GPU Evaluation}

We re-run the full pytest suite per variant on the RTX 6000 with
\texttt{VRAM\_CAP\_GIB=8} set via \texttt{tools/cap\_vram.py}, which
calls \texttt{torch.cuda.set\_per\_process\_memory\_fraction(8/24)}.
\texttt{llama.cpp} is not bound by the torch allocator, so for the
GGUF-backed variant we report natural peak VRAM and confirm it stays
under 8\,GiB.

\begin{table}[h]
\caption{Phase 3, RTX 6000, \texttt{VRAM\_CAP\_GIB=8}.}
\label{tab:rtx-capped}
\begin{tabular}{lrrr}
\toprule

The largest peak is 5.4\,GiB (the 4\,B baseline). Every other variant
uses less than 3\,GiB. Final scores match the uncapped run because the
cap only reserves VRAM and does not change numerics. The Vulkan path
used by GPT4All has the smallest footprint and provides a CUDA-free
option for non-NVIDIA silicon.

\section{Conclusion}

Four targeted changes bring TokenSmith from a server-class system to
one that runs in 40 seconds on a 4-core / 8-GiB profile without losing
answer quality. The 23\,M MiniLM variant matches the 4\,B baseline
within 1.4 points on final score; the 4\,B baseline still runs and
remains the highest-quality option for users who can spare the time.
The system is no longer bottlenecked on GPU memory at this profile;
the remaining latency floor is cold extraction, which the
\texttt{pypdfium2} fast path addresses. Code is organized as one
branch per feature on the project's git remote
(\texttt{feat/parallel-extraction}, \texttt{feat/pypdfium-extractor},
\texttt{feat/embedder-router}, \texttt{feat/pluggable-embedders}),
plus a \texttt{tools} branch and an \texttt{experiments} branch with
the per-experiment artifacts.
