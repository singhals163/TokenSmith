\begin{abstract}
TokenSmith is a local-first RAG system for textbook question answering.
The original implementation targets server-class GPUs. We re-target it
for consumer hardware (one GPU with at most 8\,GiB of memory, four CPU
cores) by making four pipeline changes: parallel PDF extraction, a
\texttt{pypdfium2} fast extraction path, a hardware-aware embedding
router with length-sorted batches, and a pluggable embedder that can
target \texttt{llama.cpp}, GPT4All (CUDA or Vulkan), or HuggingFace
\texttt{sentence-transformers} at a fixed call site. On a Quadro RTX
6000 capped at 8\,GiB, every variant stays well under the cap. A 23\,M
MiniLM embedder finishes the end-to-end pytest suite in 40 seconds at a
final score of 0.578, within 1.4 points of the 4\,B baseline (0.592)
which itself fits in 5.4\,GiB.
\end{abstract}

\keywords{retrieval-augmented generation, local inference, embeddings,
consumer GPU, FAISS}

\section{Introduction}

The original TokenSmith assumes a server with a large GPU. The goal of
this project is to make it run on a student's laptop with no external
API calls. We use the Silberschatz \emph{Database System Concepts}
textbook as the corpus (2{,}195 pages, 2{,}664 chunks) and target a
profile of one GPU with 8\,GiB of memory, four CPU cores, and 32\,GiB
of host RAM.

The pipeline has four stages: PDF extraction to markdown, indexing
into FAISS and BM25, retrieval with a cross-encoder reranker, and
generation through a 1.5\,B Qwen2.5-Instruct GGUF model on
\texttt{llama.cpp}. We make one change per stage and measure the
impact on a server (A100 / V100) and on a Quadro RTX 6000 used as the
consumer-GPU proxy.

\section{Phase 1: Parallel PDF Extraction}

The original extractor calls \texttt{docling} on the full PDF
sequentially. Page-level layout analysis is the slow part and it is
independent across pages, so we parallelize it.

\texttt{split\_pdf} divides the master PDF into fixed-size page-range
chunks using \texttt{pypdf}. \texttt{process\_pdf\_chunk} is the worker:
it imports \texttt{docling}, converts one chunk, and serializes it page
by page. A \texttt{ProcessPoolExecutor} runs the workers; results come
back keyed by start-offset and the reduce phase concatenates them in
order. The \texttt{docling} import is deferred into the worker so that
downstream callers (the indexer, pytest) do not pay its 100\,MB
cold-load cost.

\begin{table}[h]
\caption{Phase 1: docling extraction on the 2{,}195-page textbook.}
\label{tab:phase1}
\begin{tabular}{lrrr}
\toprule
host (cores) & workers & wall (s) & speedup \\
\midrule
server (64) & 1  & $\sim$1{,}930 & 1.0$\times$ \\
server (64) & 32 & $\sim$76      & \textbf{25.4$\times$} \\
RTX 6000 (4) & 4 & 499           & 3.9$\times$ \\
\bottomrule
\end{tabular}
\end{table}

Speedup saturates near 32 workers on the server because the
\texttt{pypdf} split phase becomes the Amdahl tail. On four cores we
hit the core-count ceiling.

\section{Phase 1.5: pypdfium2 Fast Path}

Even fully parallel, \texttt{docling} is overkill for plain-text
textbooks. Phase 1.5 adds a second extraction backend without layout
analysis. The new module \texttt{src/preprocessing/pypdfium\_extractor.py}
uses \texttt{pypdfium2} to read raw text from each page range. There is
no temporary split; workers seek into the master PDF directly. A short
post-processor strips table-of-contents lines and promotes
``\texttt{N.N Title}'' into ``\texttt{\#\# N.N Title}'' so the existing
markdown section splitter still works.

On the RTX 6000 host, full extraction drops from 499\,s
(\texttt{docling}) to 8.66\,s (\texttt{pypdfium2}), a 57$\times$
improvement. End-to-end final score on the regenerated index is 0.606,
slightly above the 0.592 we get on the \texttt{docling} pipeline.

\section{Phase 2: Hardware Router and Batch Sorting}

Two small changes in the embedder reduce ingestion wall time.

\paragraph{Length-sorted batches.}
\texttt{llama.cpp} pads each embedding batch to the longest text in
that batch. Random ordering wastes compute on short texts that share a
batch with one long text. We sort inputs by length descending before
batching and permute the result back after encoding. The FAISS index
ID order is unchanged.

\paragraph{Dynamic GPU / CPU routing.}
\texttt{detect\_gpu()} checks for \texttt{nvidia-smi} (CUDA) or Apple
Silicon (Metal). When a GPU is reachable the indexer takes a sequential
path with \texttt{batch\_size=32}; otherwise it uses a CPU
\texttt{multiprocessing.Pool} with the \texttt{mmap}'d GGUF shared
across workers. This avoids a CPU pool fighting the GPU's allocator on
mixed hosts.

\begin{tabular}{lrr}

The 8\,GiB cap is non-binding because \texttt{llama.cpp} is not bound
by torch's per-process memory fraction.

\section{Phase 3: Pluggable Embedding Backends}

A 4\,B embedder is heavy. Phase 3 makes the embedder swappable so we
can compare lighter alternatives at a fixed call site.

\texttt{src/embedder.py} introduces a \texttt{BaseEmbedder} ABC with
three implementations: \texttt{LlamaCppEmbedder} for GGUF models,
\texttt{GPT4AllEmbedder} for Nomic GGUFs (with \texttt{device=kompute}
for Vulkan or \texttt{device=cuda} for NVIDIA), and
\texttt{HFSentenceTransformerEmbedder} for HuggingFace
\texttt{sentence-transformers}. A factory
\texttt{make\_embedder(model\_path, backend)} returns the right
implementation and a back-compat alias keeps existing import sites
working.

Two adjacent fixes ship in this phase. We add a sentence-level BLEU-4
metric with NIST method-1 smoothing under \texttt{tests/metrics/bleu.py}.
We also fix a pollution bug in \texttt{tests/metrics/semantic.py}: the
original code set \texttt{CUDA\_VISIBLE\_DEVICES=''} at module import,
which silently disabled the GPU for any sibling process in the same
interpreter. We pin the scoring model to \texttt{device='cpu'} instead.
End-to-end pytest wall on a typical V100 run drops from $\sim$700\,s to
$\sim$45\,s.

\begin{table}[h]
\caption{Phase 3, V100, 11-question benchmark suite.}
\label{tab:phase3-v100}
\begin{tabular}{lrrrrr}
\toprule
variant & params & sem & kw & BLEU & final \\
\midrule
qwen4b              & 4\,B   & 0.761 & 0.310 & 0.026 & 0.592 \\
nomic\_st           & 137\,M & 0.735 & 0.346 & 0.032 & 0.589 \\

\end{tabular}

The 23\,M MiniLM variant slightly beats the 4\,B baseline on final
score. Its retrieval-only quality is weaker, but the cross-encoder
reranker recovers the gap as long as the right chunk is somewhere in
the top-50 FAISS neighbours.

\section{Consumer-GPU Evaluation}

We re-run the full pytest suite per variant on the RTX 6000 with
\texttt{VRAM\_CAP\_GIB=8} set via \texttt{tools/cap\_vram.py}, which
calls \texttt{torch.cuda.set\_per\_process\_memory\_fraction(8/24)}.
\texttt{llama.cpp} is not bound by the torch allocator, so for the
GGUF-backed variant we report natural peak VRAM and confirm it stays
under 8\,GiB.

\begin{table}[h]
\caption{Phase 3, RTX 6000, \texttt{VRAM\_CAP\_GIB=8}.}
\label{tab:rtx-capped}
\begin{tabular}{lrrr}
\toprule
variant & wall (s) & peak VRAM (MiB) & final \\
\midrule
minilm                       & \textbf{40.3} & 2{,}214 & 0.578 \\
nomic\_st                    & 36.2 & 2{,}706 & 0.563 \\
qwen4b                       & 37.1 & 5{,}392 & 0.592 \\
nomic\_gpt4all\_gpu (kompute) & 36.2 & \textbf{2{,}124} & 0.558 \\
\bottomrule
\end{tabular}
\end{table}

The largest peak is 5.4\,GiB (the 4\,B baseline). Every other variant
uses less than 3\,GiB. Final scores match the uncapped run because the
cap only reserves VRAM and does not change numerics. The Vulkan path
used by GPT4All has the smallest footprint and provides a CUDA-free
option for non-NVIDIA silicon.

\section{Conclusion}

Four targeted changes bring TokenSmith from a server-class system to
one that runs in 40 seconds on a 4-core / 8-GiB profile without losing
answer quality. The 23\,M MiniLM variant matches the 4\,B baseline
within 1.4 points on final score; the 4\,B baseline still runs and
remains the highest-quality option for users who can spare the time.
The system is no longer bottlenecked on GPU memory at this profile;
the remaining latency floor is cold extraction, which Phase 1.5
addresses. Code is organized as one branch per feature on the
project's git remote (\texttt{feat/parallel-extraction},
\texttt{feat/pypdfium-extractor}, \texttt{feat/embedder-router},
\texttt{feat/pluggable-embedders}), plus a \texttt{tools} branch and an
\texttt{experiments} branch with the per-experiment artifacts.
